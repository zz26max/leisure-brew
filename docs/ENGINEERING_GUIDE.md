# 闲里茶咖工程规范

> 本文档是前端、微信小程序及后续协作的长期约束。开始编码、评审或生成代码前必须先阅读；当实现与本文冲突时，优先更新本文并说明原因，不允许悄悄绕过。

## 1. 产品与工程原则

1. **先保证业务可信，再追求视觉新鲜。** 金额、身份、订单状态等规则由服务端兜底，客户端只负责表达和交互。
2. **代码写给下一位维护者看。** 选择朴素的命名、短函数和清晰的数据流，不为了“高级感”制造抽象。
3. **一次只保留一个事实来源。** 登录态、接口类型、设计变量和订单状态不能在多个文件中重复定义。
4. **错误必须可见、可恢复。** 不静默吞异常，不只写 `console.log`；用户得到可理解的提示，开发者得到必要上下文。
5. **不复制参考品牌。** 古茗、茉莉奶白仅用于提炼“鲜活日常、东方摩登、花香留白”的气质，不复制其商标、插画、包装纹样、商品图或页面布局。
6. **旧端冻结，新端迁移。** `frontend/` 与当前 `miniprogram/` 只作为行为参考，除阻塞迁移的严重问题外不再扩展功能。

## 2. 目标仓库结构

```text
leisure-brew/
├─ apps/
│  ├─ admin/                 Vue 3 门店运营台
│  └─ wechat/                原生 TypeScript 微信小程序源工程
├─ packages/
│  ├─ api-contract/          双端共享的纯 TypeScript 类型、枚举与协议
│  └─ design-tokens/         色彩、字号、间距等平台无关变量
├─ backend/                  Spring Boot 服务
├─ frontend/                 旧 Vue 2 管理端，迁移期只读
├─ miniprogram/              旧编译产物，迁移期只读
├─ docs/                     规范、决策与迁移说明
└─ docker-compose.yml
```

新代码不从旧端复制整段实现。允许复用业务规则、接口路径和经过确认的文案；复用前应先重命名、补类型并按新结构落位。

## 3. 技术基线

### 3.1 管理端

- Vue 3、TypeScript、Vite
- Vue Router、Pinia
- Element Plus：仅作为基础控件，不照搬默认后台视觉
- Axios：统一通过 `services/http` 实例访问
- ECharts：按需引入
- Vitest、Vue Test Utils
- ESLint、Prettier、Stylelint

统一使用 Composition API 和 `<script setup lang="ts">`。禁止新增 Options API、类组件、装饰器式 Vuex 或全局事件总线。

### 3.2 微信小程序

- 微信原生小程序 + TypeScript
- 原生组件、Behavior 与自定义组件
- 原生 `wx.request` 的 Promise 封装
- 不引入大型跨端框架或通用 UI 库
- 使用微信开发者工具与 `miniprogram-ci` 做构建检查

只有出现明确的支付宝、抖音或 H5 发布计划时，才重新评估 uni-app/Taro。

### 3.3 包管理与运行环境

- 新工作区统一使用 `pnpm`，只提交根目录 `pnpm-lock.yaml`。
- Node.js 使用当前维护中的 LTS 版本，并在根目录 `.nvmrc` 与 `package.json#engines` 固定主版本。
- 不混用 npm、yarn 与 pnpm；旧端锁文件在迁移完成前保留。
- Java 与 Maven 版本以 `backend/pom.xml` 和 Maven Wrapper 为准。

## 4. Git 仓库管理

### 4.1 分支

- `main`：始终可构建、可演示。
- 功能分支：`feat/<scope>-<summary>`。
- 修复分支：`fix/<scope>-<summary>`。
- 重构分支：`refactor/<scope>-<summary>`。
- 文档和工程化：`docs/...`、`chore/...`。
- 自动化代理创建分支时使用仓库约定前缀 `codex/`，例如 `codex/feat-admin-shell`。

分支名使用小写英文和短横线，不出现姓名、日期或 `final-v2` 一类无意义后缀。

### 4.2 提交

提交信息采用 Conventional Commits：

```text
feat(admin): add order workbench
fix(wechat): prevent duplicate checkout
refactor(contract): unify order status type
docs: record frontend migration rules
```

- 一次提交只表达一个意图。
- 标题使用祈使语气，不超过 72 个字符，不写“修改一些问题”。
- 代码格式化不能与大规模业务修改混在同一提交。
- 禁止提交 `.env`、密钥、证书、生产域名、真实手机号、数据库备份和个人 IDE 配置。
- 禁止提交 `dist/`、微信编译输出、测试覆盖率或临时诊断文件。
- 不重写他人已推送历史；禁止无确认执行 `reset --hard`、强推或批量删除。

### 4.3 合并请求

合并请求必须包含：

1. 为什么改，而不只是改了什么；
2. 用户可见变化截图或录屏；
3. 已执行的测试与结果；
4. 配置、数据库和兼容性影响；
5. 未解决的边界与回滚方式。

至少一位评审者确认后合并。作者需自行解决冲突并重新执行检查。

## 5. TypeScript 与代码风格

### 5.1 通用规则

- UTF-8、LF、文件末尾保留换行。
- 两个空格缩进，单引号，不写分号；由 Prettier 自动处理。
- 默认使用 `const`，确需重新赋值时才使用 `let`，禁止 `var`。
- 不使用 `any`；临时未知值使用 `unknown` 并显式收窄。
- 不使用非空断言掩盖数据问题；为缺失状态设计分支。
- 单个函数尽量不超过 40 行，组件文件尽量不超过 300 行。
- 嵌套超过三层时优先使用提前返回或提取函数。
- 删除死代码，不保留大段注释掉的旧实现。

### 5.2 命名

- 变量与函数：`camelCase`，名称表达业务含义，如 `pendingOrderCount`。
- 类型、组件：`PascalCase`。
- 常量：只对真正全局不变量使用 `UPPER_SNAKE_CASE`。
- 布尔值以 `is`、`has`、`can`、`should` 开头。
- 事件处理函数使用 `handleXxx`，纯动作函数使用动词开头，如 `loadOrders`。
- 文件夹使用 `kebab-case`；Vue 组件文件使用 `PascalCase.vue`。
- 禁止 `data1`、`objNew`、`listData`、`handleFun` 等无业务含义命名。

### 5.3 注释

注释解释“为什么”，不复述代码。业务妥协、平台限制和非直观边界需要注释；显而易见的赋值不写注释。

```ts
// 微信会重复投递支付结果，因此服务端状态更新必须具备幂等性。
await confirmPayment(orderNumber)
```

禁止带有“第一步、第二步”“老师接口”“临时先这样”等教学或过程性注释。

## 6. Vue 管理端规范

### 6.1 分层

```text
src/
├─ app/            应用启动、路由与全局 Provider
├─ assets/         自有静态资源
├─ components/     无业务或跨业务组件
├─ features/       按业务领域组织的页面、组件、store 与 service
├─ services/       HTTP、WebSocket、存储等基础能力
├─ stores/         仅全局会话和应用级状态
├─ styles/         reset、tokens、通用布局
└─ types/          仅管理端使用的公共类型
```

- 页面逻辑放在 `features/<domain>`，不建立巨大的 `views` 和 `api` 平行目录。
- 组件只通过 props、emits 或 store 通信，不直接读写其他组件实例。
- Pinia 只保存跨页面或需持久化状态；表单局部状态留在组件内。
- 派生数据使用 `computed`，不额外同步一份副本。

### 6.2 组件与可访问性

- 每个页面必须设计加载、空数据、错误和无权限状态。
- 可点击非按钮元素必须改为 `<button>` 或补齐键盘语义。
- 表单字段有可见标签；错误提示说明如何修正。
- 弹窗打开后聚焦首个可操作元素，关闭后恢复触发点。
- 颜色不是唯一状态提示；文字、图标或形状至少补充一种。
- 支持 1280px 管理端基准宽度，并保证窄屏不会遮挡核心操作。

## 7. 微信小程序规范

```text
apps/wechat/
├─ miniprogram/
│  ├─ app.ts / app.json / app.wxss
│  ├─ components/
│  ├─ pages/
│  ├─ services/
│  ├─ store/
│  ├─ styles/
│  └─ utils/
├─ project.config.json
└─ package.json
```

- 页面只负责组合和交互，网络请求集中在 `services`。
- `setData` 只更新实际变化的最小字段，不整页复制大对象。
- 所有异步操作使用 `try/catch/finally` 管理加载状态。
- 页面卸载后不继续写入状态；定时器和监听器必须清理。
- 购物袋以商品 ID + 规格组合生成稳定行项目键。
- 提交订单按钮需要客户端防重复，但服务端幂等仍是最终保障。
- 不在代码中写死 API 地址、AppID、门店电话和环境密钥。
- WXML 不承载复杂表达式，转换逻辑提前放入页面数据或纯函数。

## 8. API 与状态规范

- 共享协议放在 `packages/api-contract`，只能包含平台无关 TypeScript。
- 接口请求和响应必须有明确类型，不手写重复的局部接口。
- 金额在显示层格式化为两位小数；业务层保持服务端返回的精度，不使用浮点数自行重算订单总额。
- 日期传输使用后端约定格式，展示统一经过格式化函数。
- 订单状态使用枚举与映射表，禁止在模板中散落 `status === 2`。
- HTTP 401 清理会话并跳转登录；403 显示无权限；业务错误保留服务端可读消息。
- 仅对幂等 GET 请求自动重试，提交、支付和状态变更不自动重试。
- WebSocket 消息必须校验结构；未知事件忽略并记录开发日志，不能让页面崩溃。

## 9. 样式与品牌规范

闲里茶咖的方向是“东方鲜活茶感”，不是传统仿古，也不是奢侈品纹样。

### 9.1 基础色

```text
tea-900       #173F35  深茶绿，主文字与重点背景
tea-700       #286454  主品牌绿
tea-100       #DDECE5  轻提示与选中背景
rice-50       #FCFAF4  页面底色
rice-100      #F5F0E5  卡片与分区
jasmine-400   #E8C96A  花蕊黄，只做小面积强调
peach-400     #F1A38B  果茶与活动辅助色
ink-900       #202823  正文
ink-500       #6C756F  次级文字
line          #E5E1D7  边界
```

- 主页面以米白和茶绿为主，黄色与桃色占比不超过 10%。
- 不使用大面积纯黑、荧光渐变、玻璃拟态或通用后台蓝。
- 圆角表达杯器的柔和感：卡片 20–28px，控件 10–14px，胶囊标签使用 999px。
- 阴影轻且暖，不使用浓重悬浮阴影。

### 9.2 字体与文案

- 正文优先系统黑体；品牌标题可使用系统宋体回退，不在线加载来源不明字体。
- 标题简短、有呼吸感；正文直接说明下一步，不写“尊敬的用户”。
- 使用“购物袋、来一杯、今天想喝什么、门店伙伴”等生活化表达，但不为了可爱牺牲准确性。
- 后台文案克制专业，小程序文案轻松自然。

## 10. 测试与质量门槛

每次合并至少通过：

```text
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

关键流程还需覆盖：

- 管理端：登录、订单筛选、接单/拒单、商品上下架、员工一次性口令。
- 小程序：商品规格、购物袋、地址、提交订单、防重复点击、订单详情。
- 共享协议：状态映射、金额与日期格式化、响应结构守卫。

Bug 修复应先补能复现问题的测试，再修改实现。无法自动化时，在合并请求中写明人工验证步骤。

## 11. 安全与隐私

- Token 不写入日志、URL 查询参数或错误上报正文；WebSocket 因平台限制使用查询参数时，服务端和代理日志必须脱敏。
- 客户端不保存密码、身份证完整信息和支付敏感字段。
- 展示手机号与身份证时默认脱敏。
- HTML 字符串禁止直接插入 DOM；确需富文本时使用白名单清洗。
- 不根据前端传入的用户 ID 判断数据归属。
- 上传文件校验类型、尺寸和服务端返回值，预览 URL 不等于可信内容。

## 12. 迁移策略

1. 先完成共享协议、设计变量、登录和应用壳。
2. 管理端优先迁移订单工作台，再迁移菜单、套餐、员工和报表。
3. 小程序优先迁移菜单浏览、规格与购物袋，再迁移结算、地址和订单。
4. 每迁移一个模块就与后端真实联调，并记录旧端与新端行为差异。
5. 新端覆盖全部核心流程且连续通过验收后，才删除旧端。

迁移期间不要求新旧端 DOM 或视觉一致，只要求业务结果一致或有记录地改进。

## 13. 完成定义

一个任务只有同时满足以下条件才算完成：

- 用户目标已实现，加载、空、错误状态齐全；
- 代码符合本规范，没有新增类型逃逸和调试残留；
- 相关测试、类型检查和构建通过；
- 配置与文档已同步；
- 没有提交密钥、生成物或无关格式化；
- 评审者能从提交和说明中理解设计取舍。

## 14. 决策记录

对影响多个模块、引入新依赖或改变接口的决策，在 `docs/decisions/` 新建 ADR：

```text
0001-use-native-wechat-miniprogram.md
0002-api-contract-boundary.md
```

每份 ADR 包含背景、决定、替代方案、影响与日期。这样即使对话上下文丢失，仓库仍保留完整理由。
